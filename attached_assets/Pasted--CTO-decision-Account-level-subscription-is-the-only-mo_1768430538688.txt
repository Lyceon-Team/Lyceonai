## CTO decision: “Account-level subscription” is the only model that will stay sane at scale

You already have the right conceptual primitive in your codebase: **`account_id`** plus **`entitlements`** keyed by that account. The restructure is:

* **One Stripe subscription ↔ one `account_id`**
* **Multiple users (student + guardian) can belong to the same account**
* “Paid” means **premium entitlements for the account**, which unlock:

  * Student: unlimited study access (and any other premium study features you define)
  * Guardian: parent dashboard + calendar + KPIs **for the linked student(s)** in that account

This also solves the “student can pay” requirement cleanly: student simply pays for the account they belong to, and any linked guardian on the same account inherits premium access.

---

## Required access contract (explicit)

### Premium entitlements grant:

* **Student (role=student)**: premium study access (unlimited practice/AI/chat/etc. per your premium definition)
* **Guardian (role=guardian)**: parent dashboard + calendar + KPIs for the specific student(s) they’re linked to **within the same account**

### Premium does *not* grant:

* Guardian viewing *any* random student. Guardian still needs a valid guardian link, and the linked student must be in the same account.

---

## The two biggest bugs you must eliminate

1. **Paid subscription recorded in Stripe but DB entitlement stale**
   Fix: `/api/billing/status` must self-heal by reconciling with Stripe when entitlement is not active/trialing.

2. **Payment not linked deterministically to the correct user/student/account**
   Fix: Stripe checkout must carry `account_id` in metadata (and ideally `payer_user_id`), and webhooks must upsert entitlements by `account_id` only.

## Change 2 — Webhooks must upsert entitlements strictly by `account_id`

### File: `server/lib/webhookHandlers.ts`

Find every place you compute `accountId` from Stripe objects.
You must enforce this rule:

* `account_id = event.data.object.metadata.account_id` OR `client_reference_id`
* If missing, webhook MUST log error and mark event as failed. No “best effort” mapping by email.

#### 2A) Replace “account id resolution” with strict resolution

Insert this helper (near the top of file or inside handler file) and use it everywhere:

```ts
function requireAccountIdFromStripeObject(obj: any): string {
  const fromMeta = obj?.metadata?.account_id;
  const fromClientRef = obj?.client_reference_id;
  const accountId = fromMeta || fromClientRef;

  if (!accountId || typeof accountId !== "string") {
    throw new Error("Missing account_id on Stripe object metadata/client_reference_id");
  }
  return accountId;
}
```

Then in the webhook event handlers (checkout.session.completed, customer.subscription.updated, invoice.paid, etc.), replace any mapping logic with:

```ts
const accountId = requireAccountIdFromStripeObject(obj);
```

#### 2B) Always write these fields into entitlements

Whenever you call `upsertEntitlement(accountId, ...)`, ensure you pass:

* `stripe_customer_id`
* `stripe_subscription_id`
* `status`
* `plan`
* `current_period_end`

No partial updates.

#### 3B) Add Stripe reconciliation when entitlement is not active/trialing

Immediately after you load `entitlement` for `accountId`, insert:

```ts
  const isPaid =
    entitlement?.status === "active" || entitlement?.status === "trialing";

  // Self-heal: if DB says not paid but Stripe says active, reconcile now.
  if (!isPaid) {
    try {
      const stripeCustomerId = entitlement?.stripe_customer_id || null;
      if (stripeCustomerId) {
        const subs = await stripe.subscriptions.list({
          customer: stripeCustomerId,
          status: "all",
          limit: 10,
        });

        const best =
          subs.data.find(s => s.status === "active") ||
          subs.data.find(s => s.status === "trialing") ||
          subs.data.sort((a, b) => (b.created || 0) - (a.created || 0))[0];

        if (best) {
          const mapped = mapStripeStatusToEntitlement(best); // use your existing helper
          await upsertEntitlement(accountId, {
            plan: mapped.plan,
            status: mapped.status,
            stripe_customer_id: stripeCustomerId,
            stripe_subscription_id: best.id,
            current_period_end: best.current_period_end,
          });

          // Reload entitlement after reconciliation
          entitlement = await getOrCreateEntitlement(accountId);
        }
      }
    } catch (e) {
      // Do not throw; status endpoint should be reliable
      console.error("[BILLING] status reconcile failed", e);
    }
  }
```

This makes access deterministic even if webhook delivery fails.

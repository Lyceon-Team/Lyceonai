# Phase 1 — Make auth resolution deterministic (this is the #1 blocker)
Problem: Current production debug outputs show resolvedUserId null and hasBearerToken false.
That means the server is not extracting the user token reliably from cookies/headers.

### Requirement
Create ONE shared helper used by:
- /api/auth/debug
- ALL practice endpoints (next/answer/end-session)

Helper must:
1) Read bearer token from Authorization header (case-insensitive):
   - accept "Bearer <token>" or "bearer <token>"
2) If no Authorization header token, read cookie token from ANY cookie that contains "access-token"
   - do NOT assume only "sb-access-token"
   - Supabase often uses project-scoped cookie names; search keys.
3) If cookie value looks JSON (starts with "{" or "["), attempt JSON.parse and extract a token if present.
   - If parsing fails, ignore and continue.
4) If token cannot be found => return null userId and the practice route must return 401 with a clear error.

Implementation constraints:
- NO new packages.
- Use existing supabase client to validate token:
  supabaseAnon.auth.getUser(token)
- If supabaseAnon.auth.getUser fails => treat as unauthenticated.

Add a small diagnostic to /api/auth/debug:
- cookieKeys: list of cookie names seen (names only, no values)
- authHeaderPresent: boolean
- tokenSource: 'bearer' | 'cookie:<name>' | null
- tokenLength: number|null
- bearerParsed: boolean

Proof required (local + production-ready):
- Provide code snippet of helper.
- Provide /api/auth/debug output showing:
  cookieKeys populated when called from browser session,
  and tokenSource non-null when user is logged in.

IMPORTANT: practice endpoints must use this same helper to get userId.
DO NOT duplicate token parsing across files.

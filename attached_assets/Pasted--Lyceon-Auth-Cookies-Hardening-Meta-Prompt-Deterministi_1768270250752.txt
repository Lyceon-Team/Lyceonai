You are a senior full-stack engineer working in the Lyceon monorepo on Replit.
The current issue: login sometimes works and sometimes fails, likely due to stale/competing cookies and mixed auth authorities.
Supabase is the single source of truth for authentication.

3) The API never infers auth from legacy/custom cookies.
4) All protected API routes authenticate via verified Supabase JWT (Authorization header OR Supabase SSR session extraction), then authorize using DB role + entitlements.
5) All legacy cookies are hard-deleted safely (correct domain/path) so stale cookies cannot break routing.
6) Provide hard proofs: grep proofs, curl proofs, and a browser check list.

## Constraints
- Do NOT introduce a new auth provider.
- Do NOT add a new “server session system” (no Redis sessions, no NextAuth).
- Keep changes minimal and safe for launch.
- Do NOT break Stripe webhooks or billing.
- Ensure guardian/student routing is based on DB role, not client cache.

# Phase 1 — Cookie Audit + Legacy Cookie Kill Switch
## 1.1 Enumerate cookie names
Search the repo for cookie usage. Grep for:
- "setHeader('Set-Cookie'"
- "Set-Cookie"
- "cookie"
- "cookies."
- "document.cookie"
- "sb-" (Supabase cookie patterns)
- common legacy names: "access", "session", "token", "guardian", "role"

Create a list:
A) Supabase cookies (expected)
B) Custom cookies (must be removed or hard-deleted)
C) Legacy cookies (must be hard-deleted)

## 1.2 Implement a deterministic cookie cleanup middleware
Add server middleware that runs on every request (or at minimum for /, /login, /guardian, /dashboard, /api/*) that:
- If any cookie name exists that is NOT part of the allowed Supabase cookie set, it sets a deletion cookie for it.

Deletion rules (must be correct):
- Clear cookie with:
  - same name
  - value empty
  - Expires in the past
  - Max-Age=0
  - Path="/" (and any other known paths used historically, like "/api")
  - Domain: include both "lyceon.ai" and ".lyceon.ai" deletions if uncertain
  - Secure=true in production
  - SameSite=Lax (or None only if truly needed; prefer Lax)
- IMPORTANT: delete for BOTH host-only and dot-domain variants if you have ever changed domain scoping.

Add a constant:
ALLOWED_AUTH_COOKIE_PREFIXES = ["sb-"]  // allow Supabase cookies
ALLOWED_COOKIE_NAMES = []              // add any truly required non-auth cookies (theme, etc.)

Hard delete everything else.

Provide a note in code:
- This middleware is intended to run for 7 days after deploy to flush legacy cookies, then can be narrowed.

Success criteria:
- After visiting the site once, the browser cookie jar contains only Supabase cookies + truly necessary non-auth cookies.

# Phase 2 — Make the API Auth Deterministic (No Cookie Guessing)
## 2.1 Create a single auth middleware for API routes
Implement `requireUser(req)` for all protected API routes:
- Extract Supabase JWT in one of these supported ways:
  Option A (preferred for API): Authorization header `Bearer <token>`
  Option B (SSR cookie flow): create Supabase server client using cookies and call `auth.getUser()`

Do NOT accept any custom cookie as proof of auth.

If no valid user:
- Return 401 JSON with stable error code: `{ error: "unauthorized", code: "AUTH_REQUIRED" }`

## 2.2 Role enforcement middleware
Implement:
- `requireRole("student"|"guardian"|"admin")`

Role must be loaded from DB `profiles.role` (or your canonical role table), not from client state or cookies.

Return 403 with:
`{ error: "forbidden", code: "ROLE_REQUIRED", requiredRole: "guardian" }`

## 2.3 Entitlement enforcement middleware (paid gates)
Implement:
- `requireEntitlementActive()`
- Reads `entitlements.status` from DB (active/trialing ok; past_due/canceled denied)
- Add caching TTL 60s max (optional); must be invalidated by webhook processing if you already have that.

Return 402/403 (choose one) with stable code:
`{ error: "payment_required", code: "ENTITLEMENT_REQUIRED" }`

Success criteria:
- Every protected route uses: requireUser -> requireRole (if needed) -> requireEntitlement (if needed)

## 3.1 Eliminate client-side role assumptions
Find any frontend code that:
- reads role from localStorage

- defaults to guardian or student
- uses "isGuardian" from stale cached state

Replace with:
- On app load, call `/api/me` (or `/api/auth/me`) that returns:
  - user id
  - role
  - entitlement status
- Store that in React Query (server state), not localStorage.

## 3.2 Make /guardian routing deterministic
Current bug: session sometimes skips /login and lands in /guardian.
Fix rules:
- If user not authenticated -> redirect to /login
- If authenticated but role != guardian -> redirect to /dashboard (or student home)
- If guardian -> allow /guardian

This decision must be based on server-verified user + DB role, not cookies or client guesswork.

Add server-side protection on the /guardian route (or the API routes it depends on).

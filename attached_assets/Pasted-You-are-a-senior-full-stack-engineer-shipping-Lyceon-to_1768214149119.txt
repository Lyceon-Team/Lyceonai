### 2) Webhook reliability + idempotency + logging
Implement:
- Ensure webhook route uses RAW body middleware, before any JSON parser:
  - Only the webhook route should use `express.raw({ type: 'application/json' })`.
  - Other routes keep `express.json()`.
- Verify signature:
  - Read header `stripe-signature`
  - Use `stripe.webhooks.constructEvent(rawBody, sig, STRIPE_WEBHOOK_SECRET)`
  - On missing/invalid signature: respond 400 with a clear error and requestId
- Idempotency:
  - Add Supabase table `stripe_event_log` with UNIQUE(event_id).
  - On webhook:
    - If event_id already exists and processed_at set -> return 200 immediately
    - Else insert row, process, then update processed_at + status + any linked account ids
- Handle only required event types; ignore others with 200:
  - checkout.session.completed (primary)
  - customer.subscription.created / updated / deleted (backup sync)
  - invoice.paid (optional)
- Important: Webhook should return 200 quickly; don’t do slow network calls without timeouts.
- Add minimal structured logging with requestId/eventId.

SQL (migration):
- `stripe_event_log`:
  - id uuid pk default gen_random_uuid()
  - event_id text unique not null
  - type text not null
  - livemode boolean not null
  - payload jsonb not null
  - received_at timestamptz default now()
  - processed_at timestamptz
  - status text (e.g., "processed" | "ignored" | "failed")
  - error text
  - account_id uuid nullable
  - user_id uuid nullable

Acceptance:
- `POST /api/billing/webhook` without signature returns 400 “Missing stripe-signature”.
- Stripe Dashboard deliveries show 2xx (no HTML 404 page response).
- Replaying the same event shows “already processed” behavior and stays 200.
